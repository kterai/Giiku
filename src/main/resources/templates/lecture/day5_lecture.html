1. オブジェクト指向とは

オブジェクト指向プログラミング（Object-Oriented Programming：OOP）とは、 プログラムを「オブジェクト」と呼ばれる単位に分割し、それらを組み合わせて構築する手法です。

日常例で理解： 家を建てる時、設計図（クラス）を作ってから実際の家（オブジェクト）を建てるのと同じです。

オブジェクト指向の基本概念

クラス（Class）

オブジェクトの設計図・雛形

例：「自動車」という概念を定義する設計図

オブジェクト（Object）

クラスから作成される実際のインスタンス

例：「トヨタのプリウス」「日産のリーフ」など具体的な車

フィールド（Field）

オブジェクトの属性（データ）

例：色、エンジン排気量、燃費など

メソッド（Method）

オブジェクトの動作（振る舞い）

例：エンジン開始、ブレーキ、ウィンカー点滅など

2. オブジェクト指向3大原則

原則 説明 日常例 実用例
カプセル化
(Encapsulation) 		データと処理を一体化し、外部から直接触れないようにする
なぜ重要？ データの整合性を保ち、予期しない変更を防ぐため
銀行のATM：暗証番号は隠され、正当な操作でのみ口座にアクセス可能 ユーザー管理システム、商品在庫管理、個人情報保護
継承
(Inheritance) 		親クラスの機能を子クラスが引き継ぐ
なぜ重要？ コードの重複を避け、保守性を向上させるため
動物→犬→柴犬：上の階層の特徴を下の階層が引き継ぐ UI部品の共通化、ゲームキャラクター、文書テンプレート
ポリモーフィズム
(Polymorphism) 		同じ命令でも、対象によって異なる動作をする
なぜ重要？ 柔軟で拡張しやすいプログラムを作るため
「鳴け」という命令で、犬は"ワン"、猫は"ニャー"と異なる反応 支払い処理（クレジット/現金/電子マネー）、ファイル処理

講師向け指導ポイント

  • • 各原則は独立しているが、実際のプログラムでは相互に関連することを強調
  • • 日常例から徐々にプログラムの例に移行すると理解しやすい
  • • 「なぜその原則が必要なのか」を必ず説明してから「どうやって実装するか」を教える
  • • 受講者に身近な例（スマホアプリ、ゲーム等）で説明すると興味を引きやすい

3. 手続き型 vs オブジェクト指向

比較項目 手続き型プログラミング オブジェクト指向プログラミング
データ管理 グローバル変数で共有
⚠️ どこからでもアクセス可能で危険
 オブジェクト内で管理
✅ 適切な制御でデータを保護
処理の流れ 上から下へ順次実行
⚠️ 複雑になると理解困難
 オブジェクト間のメッセージ交換
✅ モジュール化で理解しやすい
コード再利用 関数の呼び出し
⚠️ コピー&ペーストが多発
 継承とポリモーフィズム
✅ 効率的な再利用が可能
保守・拡張 変更の影響範囲が不明
⚠️ 1箇所の変更が全体に影響
 変更影響を局所化
✅ 安全に機能追加・修正可能

手続き型の問題例 

// グローバル変数（危険）
double radius = 5.0;
double area;

// 関数がバラバラに存在
double calculateCircleArea(double r) {
    return Math.PI * r * r;
}

void printArea() {
    area = calculateCircleArea(radius);
    System.out.println("面積: " + area);
}

問題：radius変数が誰からでも変更可能で、データの整合性が保てない

オブジェクト指向の解決例 

// クラスでデータと処理を一体化
class Circle {
    private double radius; // データ隠蔽
    
    public Circle(double radius) {
        this.radius = radius;
    }
    
    public double getArea() {
        return Math.PI * radius * radius;
    }
    
    public void printArea() {
        System.out.println("面積: " + getArea());
    }
}

解決：radiusが保護され、関連処理がまとまって管理しやすい

4. クラスとオブジェクト

建築に例えると...

🏗️ クラス = 設計図

  • • 間取り、構造、設備の仕様
  • • 何度でも使い回し可能
  • • 実際の家ではない

🏠 オブジェクト = 実際の家

  • • 設計図から建てられた具体的な家
  • • 同じ設計図から複数の家を建設可能
  • • それぞれ住所や家具が異なる

クラスの基本構造

public class Car {
    // フィールド（属性・データ）
    private String color;        // 色
    private String maker;        // メーカー
    private int engineSize;      // エンジン排気量
    private boolean isRunning;   // エンジン稼働状態
    
    // コンストラクタ（オブジェクト作成時の初期設定）
    public Car(String color, String maker, int engineSize) {
        this.color = color;
        this.maker = maker;
        this.engineSize = engineSize;
        this.isRunning = false;  // 初期状態は停止
    }
    
    // メソッド（動作・振る舞い）
    public void startEngine() {
        if (!isRunning) {
            isRunning = true;
            System.out.println(maker + "の" + color + "の車のエンジンが始動しました");
        }
    }
    
    public void stopEngine() {
        if (isRunning) {
            isRunning = false;
            System.out.println("エンジンを停止しました");
        }
    }
    
    // ゲッター（フィールドの値を取得）
    public String getColor() {
        return color;
    }
    
    public String getMaker() {
        return maker;
    }
}

オブジェクトの作成と使用

public class Main {
    public static void main(String[] args) {
        // オブジェクトの作成（インスタンス化）
        Car myCar = new Car("赤", "トヨタ", 2000);
        Car yourCar = new Car("青", "日産", 1500);
        
        // メソッドの呼び出し
        myCar.startEngine();    // トヨタの赤の車のエンジンが始動しました
        yourCar.startEngine();  // 日産の青の車のエンジンが始動しました
        
        // フィールドの値を取得
        System.out.println("私の車の色: " + myCar.getColor());  // 赤
        System.out.println("あなたの車のメーカー: " + yourCar.getMaker());  // 日産
    }
}

重要ポイント

  • • 1つのクラスから複数のオブジェクトを作成できる
  • • 各オブジェクトは独立したデータを持つ
  • • 同じメソッドでも、オブジェクトごとに異なる結果になる
  • • new演算子でオブジェクトを作成（インスタンス化）

5. カプセル化の実装

カプセル化とは？

カプセル化（Encapsulation）は、オブジェクトのデータ（フィールド）を外部から直接アクセスできないように隠蔽し、 メソッドを通じてのみアクセスできるようにする技術です。

薬のカプセルと同じ： 中の薬（データ）を直接触らせず、適切なタイミングで適切な量だけ効果を発揮させる

アクセス修飾子

修飾子 アクセス範囲 用途 実用例
private 同じクラス内のみ データ隠蔽（フィールドに使用） パスワード、個人情報、内部計算用変数
public どこからでも 外部との接点（メソッドに使用） ユーザー名取得、データ更新処理
protected 同じパッケージ＋サブクラス 継承関係での共有 親クラスの共通機能
（省略） 同じパッケージ内 パッケージ内での共有 関連クラス間のデータ共有

ゲッター・セッターメソッド

❌ 悪い例：直接アクセス

public class BankAccount {
    public int balance; // 危険！直接変更可能
}

// 使用例
BankAccount account = new BankAccount();
account.balance = -1000000; // 残高マイナス！？
account.balance = 999999999; // 不正に大金を設定！？
問題：残高を直接変更でき、データの整合性が保てない

✅ 良い例：カプセル化

public class BankAccount {
    private int balance; // 隠蔽
    
    // ゲッター：値を取得
    public int getBalance() {
        return balance;
    }
    
    // セッター：値を設定（検証付き）
    public void deposit(int amount) {
        if (amount > 0) {
            balance += amount;
        }
    }
    
    public boolean withdraw(int amount) {
        if (amount > 0 && balance >= amount) {
            balance -= amount;
            return true;
        }
        return false; // 残高不足
    }
}
解決：適切な検証を経てのみ残高を変更、データの整合性を保持

講師向け指導ポイント

  • • 「なぜprivateにするのか」を必ず説明（セキュリティ、データ整合性）
  • • 銀行口座、ゲームのHP、商品在庫など身近な例で説明すると理解しやすい
  • • ゲッター・セッターは単純な取得・設定だけでなく、検証やログ出力も可能と説明
  • • IDEの自動生成機能も紹介し、実務での効率化も伝える

6. thisキーワード

thisとは？

thisは「現在のオブジェクト自身」を指すキーワードです。 主にフィールドとパラメータの名前が同じ場合の区別や、メソッドチェーンで使用します。

鏡を見るのと同じ： 「この私の髪の色」「この私の身長」のように、自分自身を明確に指し示す

thisの使用例

1. フィールドとパラメータの区別

❌ this を使わない場合
public class Person {
    private String name;
    private int age;
    
    public Person(String name, int age) {
        name = name;  // パラメータ同士の代入！
        age = age;    // フィールドに値が設定されない
    }
}

フィールドに値が設定されず、nullや0のまま

✅ this を使った場合
public class Person {
    private String name;
    private int age;
    
    public Person(String name, int age) {
        this.name = name;  // this.name = フィールド
        this.age = age;    // name, age = パラメータ
    }
}

フィールドに正しく値が設定される

2. メソッドチェーン

public class Calculator {
    private int value;
    
    public Calculator add(int num) {
        this.value += num;
        return this;  // 自分自身を返す
    }
    
    public Calculator multiply(int num) {
        this.value *= num;
        return this;
    }
    
    public int getValue() {
        return this.value;
    }
}

// 使用例：メソッドチェーン
Calculator calc = new Calculator();
int result = calc.add(5).multiply(3).add(2).getValue();  // (5 * 3) + 2 = 17

3. 他のコンストラクタの呼び出し

public class Car {
    private String color;
    private String maker;
    private int engineSize;
    
    // メインコンストラクタ
    public Car(String color, String maker, int engineSize) {
        this.color = color;
        this.maker = maker;
        this.engineSize = engineSize;
    }
    
    // 簡易コンストラクタ（デフォルト値を使用）
    public Car(String color, String maker) {
        this(color, maker, 1500);  // メインコンストラクタを呼び出し
    }
    
    // 最簡易コンストラクタ
    public Car() {
        this("白", "トヨタ", 1500);  // メインコンストラクタを呼び出し
    }
}

thisキーワードの注意点

  • • staticメソッド内では使用不可（インスタンスが存在しないため）
  • • this()は必ずコンストラクタの最初の行に記述
  • • 名前の衝突がない場合はthis省略可能だが、明示的に書く方が分かりやすい
  • • メソッドチェーンは便利だが、デバッグが困難になる場合もある

📝 演習問題

演習1: 学生クラスの設計

以下の仕様を満たすStudentクラスを設計してください。

  • フィールド：学籍番号(studentId)、名前(name)、年齢(age)
  • すべてのフィールドはprivateで宣言
  • 適切なコンストラクタ、ゲッター、セッターを実装
  • 年齢は0以上の値のみ設定可能
  • 学生情報を表示するdisplayInfo()メソッドを実装

回答例：

public class Student {
    // フィールド（すべてprivate）
    private String studentId;
    private String name;
    private int age;
    
    // コンストラクタ
    public Student(String studentId, String name, int age) {
        this.studentId = studentId;
        this.name = name;
        setAge(age);  // セッターを使用して年齢チェック
    }
    
    // ゲッター
    public String getStudentId() {
        return studentId;
    }
    
    public String getName() {
        return name;
    }
    
    public int getAge() {
        return age;
    }
    
    // セッター
    public void setStudentId(String studentId) {
        this.studentId = studentId;
    }
    
    public void setName(String name) {
        this.name = name;
    }
    
    public void setAge(int age) {
        if (age >= 0) {  // 年齢チェック
            this.age = age;
        } else {
            System.out.println("年齢は0以上である必要があります");
        }
    }
    
    // 学生情報表示
    public void displayInfo() {
        System.out.println("学籍番号: " + studentId);
        System.out.println("名前: " + name);
        System.out.println("年齢: " + age + "歳");
    }
}
ポイント：
  • thisキーワードでフィールドとパラメータを区別
  • setAge()で入力値検証を実装
  • カプセル化によりデータの整合性を保護

演習2: 銀行口座クラスの実装

カプセル化を活用したBankAccountクラスを実装してください。

  • フィールド：口座番号(accountNumber)、口座名義(accountHolder)、残高(balance)
  • 残高は外部から直接変更不可
  • 預金メソッド：deposit(int amount) - 正の値のみ受付
  • 出金メソッド：withdraw(int amount) - 残高不足の場合はfalseを返す
  • 残高照会メソッド：getBalance()

回答例：

public class BankAccount {
    private String accountNumber;
    private String accountHolder;
    private int balance;
    
    public BankAccount(String accountNumber, String accountHolder, int initialBalance) {
        this.accountNumber = accountNumber;
        this.accountHolder = accountHolder;
        this.balance = initialBalance >= 0 ? initialBalance : 0;
    }
    
    // 預金（正の値のみ）
    public boolean deposit(int amount) {
        if (amount > 0) {
            this.balance += amount;
            System.out.println(amount + "円を預金しました。残高: " + balance + "円");
            return true;
        } else {
            System.out.println("預金額は正の値である必要があります");
            return false;
        }
    }
    
    // 出金（残高チェック付き）
    public boolean withdraw(int amount) {
        if (amount > 0 && balance >= amount) {
            this.balance -= amount;
            System.out.println(amount + "円を出金しました。残高: " + balance + "円");
            return true;
        } else if (amount <= 0) {
            System.out.println("出金額は正の値である必要があります");
            return false;
        } else {
            System.out.println("残高不足です。現在の残高: " + balance + "円");
            return false;
        }
    }
    
    // 残高照会
    public int getBalance() {
        return balance;
    }
    
    // 口座情報表示
    public void displayAccountInfo() {
        System.out.println("口座番号: " + accountNumber);
        System.out.println("口座名義: " + accountHolder);
        System.out.println("現在残高: " + balance + "円");
    }
}
カプセル化のメリット：
  • 残高の直接変更を防ぎ、不正な操作を排除
  • 預金・出金時の検証処理を確実に実行
  • 将来的な機能追加（手数料計算、履歴記録等）が容易

演習3: メソッドチェーンの実装

thisキーワードを活用してメソッドチェーンが可能なStringBuilder風のクラスを作成してください。

  • 文字列を内部で管理するStringBuilderクラス
  • append(String text) - 文字列追加（thisを返す）
  • delete(int length) - 末尾から指定文字数削除（thisを返す）
  • toString() - 現在の文字列を返す
  • 使用例：builder.append("Hello").append(" ").append("World").toString()

回答例：

public class MyStringBuilder {
    private String text;
    
    public MyStringBuilder() {
        this.text = "";
    }
    
    public MyStringBuilder(String initialText) {
        this.text = initialText != null ? initialText : "";
    }
    
    // 文字列追加（thisを返してメソッドチェーン可能に）
    public MyStringBuilder append(String str) {
        if (str != null) {
            this.text += str;
        }
        return this;  // 自分自身を返す
    }
    
    // 末尾から指定文字数削除
    public MyStringBuilder delete(int length) {
        if (length > 0 && length <= text.length()) {
            this.text = text.substring(0, text.length() - length);
        }
        return this;  // 自分自身を返す
    }
    
    // 現在の文字列を取得
    @Override
    public String toString() {
        return text;
    }
    
    // 長さを取得
    public int length() {
        return text.length();
    }
}

// 使用例
public class Main {
    public static void main(String[] args) {
        MyStringBuilder builder = new MyStringBuilder();
        
        // メソッドチェーンで連続処理
        String result = builder
            .append("Hello")
            .append(" ")
            .append("World")
            .append("!")
            .delete(1)  // "!"を削除
            .append("?")
            .toString();
            
        System.out.println(result);  // "Hello World?"
    }
}
thisの活用ポイント：
  • return this;で自分自身を返し、メソッドチェーンを実現
  • fluent interface（流暢なインターフェース）パターンの実装
  • コードの可読性と記述効率の向上

🧠 理解度クイズ

クイズ1: オブジェクト指向の基本概念

以下のコードについて、正しい説明を選んでください。 

Car myCar = new Car("赤", "トヨタ");

正解：b) Carはクラス、myCarはオブジェクト（インスタンス）である

解説：

  • Car：クラス名（設計図）
  • myCar：変数名（オブジェクトへの参照）
  • new Car("赤", "トヨタ")：実際のオブジェクト（インスタンス）を作成
  • クラスは1つでも、そこから複数のオブジェクトを作成可能

クイズ2: カプセル化の理解

以下のクラス設計のうち、適切なカプセル化が実装されているものはどれですか？

正解：b) パターンB

解説：

  • パターンA：すべてpublic→データが外部から直接変更可能で危険
  • パターンB：フィールドをprivateで隠蔽し、メソッドで制御
  • setAge()で年齢の妥当性チェックを実装
  • 必要に応じてゲッター・セッターで適切にアクセス制御

クイズ3: thisキーワードの使用

以下のコンストラクタの動作について、正しい説明を選んでください。 

public class Person {
    private String name;
    private int age;
    
    public Person(String name, int age) {
        name = name;
        age = age;
    }
}

正解：b) フィールドに値が設定されず、初期値のままになる

解説：

  • name = name;は「パラメータのname = パラメータのname」の意味
  • フィールドのnameには一切値が設定されない
  • 正しくはthis.name = name;と記述する必要がある
  • thisを使わないと、フィールドとパラメータを区別できない

修正版：

public Person(String name, int age) {
    this.name = name;  // フィールド = パラメータ
    this.age = age;    // フィールド = パラメータ
}
