1. CI/CD（継続的インテグレーション・デプロイメント）の完全理解

CI/CDとは？

CI/CD（Continuous Integration / Continuous Deployment）とは、 コードの変更を自動的にテスト・ビルド・デプロイするプロセスです。 まるで工場の自動化ラインのように、各工程（テスト・ビルド・デプロイ）を自動化して 品質を保ちながら迅速にリリースします。

CI/CD vs 従来手法 詳細比較

項目 従来手法（手動） CI/CD（自動化） 効果・メリット
デプロイ頻度 週1回〜月1回 日数回〜時間単位 市場対応速度の大幅向上
エラー検出 本番環境で発見 開発段階で自動検出 修正コスト90%削減
作業時間 半日〜1日 5〜15分 開発者の集中時間確保
品質安定性 人的ミスのリスク 一定品質を保証 障害率70%削減

GitHub Actions CI/CDパイプライン実装例

実際のWebアプリケーション用の完全なCI/CDワークフロー設定例：

# .github/workflows/ci-cd.yml
name: CI/CD Pipeline

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v3
    
    - name: Setup Node.js
      uses: actions/setup-node@v3
      with:
        node-version: '18'
        cache: 'npm'
    
    - name: Install dependencies
      run: npm ci
    
    - name: Run tests
      run: npm test
    
    - name: Run linting
      run: npm run lint
    
    - name: Security audit
      run: npm audit --audit-level high

  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v3
    
    - name: Build Docker image
      run: |
        docker build -t myapp:${{ github.sha }} .
        
    - name: Login to Docker Hub
      uses: docker/login-action@v2
      with:
        username: ${{ secrets.DOCKER_USERNAME }}
        password: ${{ secrets.DOCKER_PASSWORD }}
    
    - name: Push Docker image
      run: |
        docker tag myapp:${{ github.sha }} myapp:latest
        docker push myapp:${{ github.sha }}
        docker push myapp:latest

  deploy:
    needs: build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
    - name: Deploy to AWS ECS
      uses: aws-actions/amazon-ecs-deploy-task-definition@v1
      with:
        task-definition: task-definition.json
        service: myapp-service
        cluster: myapp-cluster

2. Docker コンテナ技術の実践マスター

Dockerとは？

Docker（ドッカー）とは、 アプリケーションとその実行環境を「コンテナ」という軽量な仮想環境にパッケージ化する技術です。 まるで引っ越し用のコンテナのように、アプリケーションに必要な全てを 1つのパッケージにまとめて、どこでも同じように動作させることができます。

Docker vs 従来技術 詳細比較

従来の仮想マシン

  • 重い（GB単位）
  • 起動に数分
  • リソース消費大
  • OS全体が必要

Dockerコンテナ

  • 軽量（MB単位）
  • 起動は数秒
  • リソース効率的
  • アプリレイヤーのみ

Dockerfile 実装例（Node.js Webアプリ）

セキュリティを考慮した本番環境対応のDockerfile：

# Dockerfile
# マルチステージビルドでセキュリティ強化
FROM node:18-alpine AS builder

# セキュリティ：非rootユーザーで実行
RUN addgroup -g 1001 -S nodejs
RUN adduser -S nextjs -u 1001

# 作業ディレクトリ設定
WORKDIR /app

# 依存関係ファイルのコピー（キャッシュ効率化）
COPY package*.json ./

# 依存関係インストール（開発用除外）
RUN npm ci --only=production && npm cache clean --force

# アプリケーションコードコピー
COPY --chown=nextjs:nodejs . .

# アプリケーションビルド
RUN npm run build

# 本番用最小イメージ
FROM node:18-alpine AS runner

# セキュリティパッケージ更新
RUN apk update && apk upgrade && apk add --no-cache dumb-init

# 非rootユーザー作成
RUN addgroup -g 1001 -S nodejs
RUN adduser -S nextjs -u 1001

WORKDIR /app

# 必要なファイルのみコピー（最小化）
COPY --from=builder --chown=nextjs:nodejs /app/node_modules ./node_modules
COPY --from=builder --chown=nextjs:nodejs /app/dist ./dist
COPY --from=builder --chown=nextjs:nodejs /app/package.json ./package.json

# 非rootユーザーに切り替え
USER nextjs

# ヘルスチェック追加
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# ポート公開
EXPOSE 3000

# セキュアな起動（dumb-initでプロセス管理）
ENTRYPOINT ["dumb-init", "--"]
CMD ["node", "dist/server.js"]

3. Kubernetes コンテナオーケストレーション

Kubernetesとは？

Kubernetes（クーベルネティス）とは、 複数のコンテナを自動で管理・運用する仕組みです。 まるでオーケストラの指揮者のように、多数のコンテナ（楽器奏者）を 調和よく動作させ、負荷に応じて自動でスケーリングします。

Kubernetes主要コンポーネント

Pod（ポッド）

最小デプロイ単位。1つ以上のコンテナをグループ化

例：Webアプリ + サイドカーログ収集

Deployment（デプロイメント）

Pod の作成・更新・スケーリングを管理

例：3つのWebサーバーPodを常時稼働

Service（サービス）

Pod への安定したネットワークアクセスを提供

例：ロードバランサー + 内部DNS

Ingress（イングレス）

外部からのHTTPS/HTTPアクセスをルーティング

例：ドメイン別にサービス振り分け

Kubernetes YAML設定例（Webアプリデプロイ）

完全なWebアプリケーションデプロイ用のKubernetesマニフェスト：

# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp-deployment
  labels:
    app: webapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: webapp
  template:
    metadata:
      labels:
        app: webapp
    spec:
      containers:
      - name: webapp
        image: myapp:latest
        ports:
        - containerPort: 3000
        env:
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: webapp-secrets
              key: database-url
        resources:
          requests:
            memory: "128Mi"
            cpu: "100m"
          limits:
            memory: "256Mi"
            cpu: "200m"
        livenessProbe:
          httpGet:
            path: /health
            port: 3000
          initialDelaySeconds: 30
          periodSeconds: 10
        readinessProbe:
          httpGet:
            path: /ready
            port: 3000
          initialDelaySeconds: 5
          periodSeconds: 5

---
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: webapp-service
spec:
  selector:
    app: webapp
  ports:
    - protocol: TCP
      port: 80
      targetPort: 3000
  type: LoadBalancer

---
# ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: webapp-ingress
  annotations:
    kubernetes.io/ingress.class: nginx
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  tls:
  - hosts:
    - myapp.example.com
    secretName: webapp-tls
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: webapp-service
            port:
              number: 80

4. Infrastructure as Code（IaC）実践

Infrastructure as Codeとは？

Infrastructure as Code（IaC：アイアック）とは、 サーバー・ネットワーク・データベースなどのインフラ構成を「コード」として管理する手法です。 まるで建築設計図のように、インフラの設計を文書化し、 自動で構築・管理できます。

主要IaCツール比較

ツール 得意分野 記述方式 実用例 学習コスト
Terraform マルチクラウド対応 HCL（宣言型） AWS/Azure/GCP統合管理 中
Ansible サーバー設定管理 YAML（手続き型） アプリデプロイ・設定変更 低
Pulumi プログラマフレンドリー TypeScript/Python 複雑なロジック実装 高
CloudFormation AWS特化 JSON/YAML AWS完全統合 中

Terraform実装例（AWS基盤構築）

VPC、サブネット、セキュリティグループ、ECS Fargate、ALB、RDSを含む完全なAWSインフラ：

# main.tf
terraform {
  required_version = ">= 1.0"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = var.aws_region
}

# VPC
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = {
    Name = "${var.project_name}-vpc"
  }
}

# Public Subnets
resource "aws_subnet" "public" {
  count             = 2
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.${count.index + 1}.0/24"
  availability_zone = data.aws_availability_zones.available.names[count.index]
  
  map_public_ip_on_launch = true

  tags = {
    Name = "${var.project_name}-public-${count.index + 1}"
  }
}

# Private Subnets
resource "aws_subnet" "private" {
  count             = 2
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.${count.index + 10}.0/24"
  availability_zone = data.aws_availability_zones.available.names[count.index]

  tags = {
    Name = "${var.project_name}-private-${count.index + 1}"
  }
}

# Internet Gateway
resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id

  tags = {
    Name = "${var.project_name}-igw"
  }
}

# Application Load Balancer
resource "aws_lb" "main" {
  name               = "${var.project_name}-alb"
  internal           = false
  load_balancer_type = "application"
  security_groups    = [aws_security_group.alb.id]
  subnets            = aws_subnet.public[*].id

  enable_deletion_protection = false
}

# ECS Cluster
resource "aws_ecs_cluster" "main" {
  name = "${var.project_name}-cluster"

  setting {
    name  = "containerInsights"
    value = "enabled"
  }
}

# RDS Aurora Cluster
resource "aws_rds_cluster" "main" {
  cluster_identifier      = "${var.project_name}-aurora"
  engine                  = "aurora-mysql"
  engine_version         = "8.0.mysql_aurora.3.02.0"
  availability_zones     = data.aws_availability_zones.available.names
  database_name          = var.database_name
  master_username        = var.database_username
  master_password        = var.database_password
  backup_retention_period = 5
  preferred_backup_window = "07:00-09:00"
  
  vpc_security_group_ids = [aws_security_group.rds.id]
  db_subnet_group_name   = aws_db_subnet_group.main.name
  
  storage_encrypted = true
  skip_final_snapshot = true
}

5. DevOps文化と監視システム

DevOps文化とは？

DevOps（デブオプス）とは、 Development（開発）とOperations（運用）の協働により、 品質とスピードを両立させる文化・手法です。 まるでレーシングチームのように、開発者・運用者・QAが 一丸となって最高のパフォーマンスを目指します。

DevOps成熟度指標（DORA Metrics）

デプロイ頻度

どれくらい頻繁にリリースするか

目標: 日単位〜週単位

リードタイム

コミットから本番反映までの時間

目標: 1時間〜1日

平均復旧時間

障害発生から復旧までの時間

目標: 1時間以内

変更障害率

デプロイに起因する障害の割合

目標: 5%以下

理解度クイズ

問題1: CI/CDパイプラインの効果

CI/CDパイプラインを導入することで期待される効果として、最も適切でないものを選んでください。

A) デプロイ頻度の向上（週1回 → 日数回）
B) エラー検出の早期化（本番 → 開発段階）
C) プロジェクト管理の完全自動化
D) 手動作業時間の短縮（半日 → 15分）

正解: C) プロジェクト管理の完全自動化

CI/CDはコードのビルド・テスト・デプロイを自動化しますが、プロジェクト管理（計画・進捗管理・チームコミュニケーション）は別の領域です。

問題2: Kubernetes vs Docker

KubernetesとDockerの関係性について、最も正確な説明を選んでください。

A) KubernetesはDockerの上位互換である
B) Dockerでコンテナを作り、Kubernetesで複数コンテナを管理する
C) KubernetesとDockerは競合する技術である
D) KubernetesはDockerの軽量版である

正解: B) Dockerでコンテナを作り、Kubernetesで複数コンテナを管理する

Dockerはコンテナ作成技術、Kubernetesはコンテナオーケストレーション（管理・運用）技術です。相互補完の関係にあります。

問題3: DevOps成熟度指標

DORA Metricsの4つの指標に含まれない項目を選んでください。

A) デプロイ頻度
B) リードタイム
C) コードカバレッジ
D) 平均復旧時間

正解: C) コードカバレッジ

DORA Metricsは「デプロイ頻度」「リードタイム」「平均復旧時間」「変更障害率」の4つです。コードカバレッジは含まれません。
