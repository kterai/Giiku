Day 26: データベース基礎

関係モデル、ER図、SQL基本構文 - データベース設計と操作の基礎を学ぶ

学習目標

  • 関係データベースの基本概念（リレーション、タプル、属性、キー）を理解する
  • ER図を使ったデータベース設計の基礎を習得する
  • 正規化理論（第1～第3正規形）を理解し、適用できる
  • 基本的なSQL文（SELECT, INSERT, UPDATE, DELETE）を記述できる

達成基準

  • データベースの種類と特徴を説明できる
  • 関係モデルの基本要素を正しく理解している
  • 簡単なER図を描いて正規化を行うことができる
  • 基本的なSQLクエリを作成・実行できる

詳細スケジュール

前日のネットワーク応用の復習と本日のデータベース学習の概要説明。データベースの重要性と基本情報技術者試験での出題傾向を確認します。

データベースの種類（階層型、ネットワーク型、関係型、NoSQL）と特徴、関係モデルの基本概念（リレーション、タプル、属性、主キー、外部キー）について学習します。
講義 - データベースの基本理論

実際のテーブル例を使って、リレーション、タプル、属性の概念を理解し、主キーと外部キーの役割を確認する演習を行います。
演習 - 関係モデルの理解

休憩時間です。昼食をとり、リフレッシュしましょう。

ER図（Entity-Relationship図）の読み方と書き方、エンティティ、アトリビュート、リレーションシップの概念、データベース設計の基本手順を学習します。
講義 - データベース設計手法

学校管理システムや図書館システムなどの具体例を使って、実際にER図を作成する演習を行います。要件定義からテーブル設計まで一連の流れを体験します。
演習 - ER図作成実践

データベース正規化の目的と手法、第1正規形・第2正規形・第3正規形の概念と適用方法、正規化によるデータの冗長性排除について学習します。
講義 - 正規化理論の基礎

短い休憩時間です。リフレッシュしましょう。

SQL（Structured Query Language）の基本構文、SELECT文によるデータ抽出、INSERT・UPDATE・DELETE文によるデータ操作、WHERE句による条件指定を学習します。
演習 - SQL基本操作

データベース基礎分野の理解度を確認するテストを実施します。関係モデル、ER図、正規化、基本SQL文に関する問題に取り組みます。
テスト - データベース基礎

本日の学習内容の振り返りと質疑応答。データベース設計や正規化でよくある疑問点について解説し、明日のデータベース応用学習への準備を行います。

データベース基礎

データベースの種類と特徴

階層型データベース

データを木構造で管理するデータベース。親子関係が明確で、IBMのIMSが代表例です。

  • 特徴: 高速アクセス、構造が単純
  • 欠点: 柔軟性に欠ける、データの冗長性

ネットワーク型データベース

複雑な関係をネットワーク構造で表現するデータベース。CODASYLが標準化しました。

  • 特徴: 多対多の関係を直接表現
  • 欠点: 構造が複雑、プログラミングが困難

関係型データベース（RDB）

データを表（テーブル）の形で管理する現在最も普及しているデータベース。

  • 特徴: 直感的な構造、SQLによる標準的な操作
  • 利点: データの独立性、正規化による冗長性排除

NoSQLデータベース

関係型以外の新しいデータベース技術の総称。ビッグデータ処理に適しています。

  • 種類: ドキュメント指向、キーバリュー型、カラム指向、グラフ型
  • 特徴: スケーラビリティ、柔軟なスキーマ

関係モデルの基本概念

リレーション（関係・表）

データを格納する表のこと。行と列で構成され、各列は特定の属性を表します。

  • 例: 学生テーブル、科目テーブル
  • 特徴: 重複する行がない

タプル（組・行）

リレーション内の各行のこと。一つのエンティティ（実体）を表現します。

  • 例: 一人の学生の情報
  • 特徴: 各タプルは一意に識別可能

属性（列）

リレーション内の各列のこと。エンティティの特性を表現します。

  • 例: 学生ID、氏名、年齢
  • 特徴: 各属性には定義域がある

キー

タプルを一意に識別するための属性または属性の組み合わせ。

  • 主キー: 主要な識別子
  • 外部キー: 他のテーブルとの関連付け

ER図によるデータベース設計

基本的なER図の要素

エンティティ

データとして管理したい実体
（学生、科目、教員など）

アトリビュート

エンティティの性質や特徴
（学生ID、氏名、年齢など）

リレーションシップ

エンティティ間の関係
（履修、担当、所属など）

設計のポイント:
  • 業務要件を正確に分析し、必要なエンティティを洗い出す
  • エンティティ間の関係性（1対1、1対多、多対多）を明確にする
  • 主キーと外部キーを適切に設定する

正規化理論

第1正規形（1NF）

条件: 各セルに単一の値のみを格納する（多値属性の排除）

正規化前:
学生ID | 氏名 | 履修科目
S001 | 田中 | 数学,英語,物理

正規化後:
学生ID | 氏名 | 履修科目
S001 | 田中 | 数学
S001 | 田中 | 英語
S001 | 田中 | 物理

第2正規形（2NF）

条件: 1NFかつ、非キー属性が主キー全体に完全従属する（部分従属の排除）

複合キーの一部にのみ依存する属性を別テーブルに分割します。

第3正規形（3NF）

条件: 2NFかつ、非キー属性間の推移従属がない（推移従属の排除）

キー以外の属性に依存する属性を別テーブルに分割します。

SQL基本構文

SELECT文（データ抽出）

-- 基本的なSELECT文
SELECT 氏名, 年齢 FROM 学生;

-- 条件付きSELECT文
SELECT * FROM 学生
WHERE 年齢 >= 20;

-- 並び替え
SELECT * FROM 学生
ORDER BY 年齢 DESC;

INSERT文（データ挿入）

-- 新しいレコードの挿入
INSERT INTO 学生
(学生ID, 氏名, 年齢)
VALUES ('S001', '田中', 20);

-- 複数レコードの挿入
INSERT INTO 学生 VALUES
('S002', '佐藤', 21),
('S003', '山田', 19);

UPDATE文（データ更新）

-- 特定レコードの更新
UPDATE 学生
SET 年齢 = 21
WHERE 学生ID = 'S001';

-- 複数列の更新
UPDATE 学生
SET 氏名 = '田中太郎', 年齢 = 22
WHERE 学生ID = 'S001';

DELETE文（データ削除）

-- 特定レコードの削除
DELETE FROM 学生
WHERE 学生ID = 'S001';

-- 条件に合致する複数レコード削除
DELETE FROM 学生
WHERE 年齢 < 18;

-- 全レコード削除
DELETE FROM 学生;
