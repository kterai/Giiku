テーマ詳細

例外処理の基本

例外処理は、プログラムの実行中に発生する予期しないエラー状況を検出し、適切に対処するための仕組みです。 例外処理により、プログラムの堅牢性が向上し、エラー発生時にもグレースフルに対応できるようになります。

例外クラスの階層構造
  • Throwable: すべての例外とエラーの基底クラス
    • Error: 深刻なエラー（通常キャッチする必要なし）
      • OutOfMemoryError
      • StackOverflowError
      • VirtualMachineError
    • Exception: プログラムで処理可能な例外
      • RuntimeException: 非チェック例外
        • NullPointerException
        • ArrayIndexOutOfBoundsException
        • IllegalArgumentException
        • ArithmeticException
      • その他の例外: チェック例外
        • IOException
        • SQLException
        • ClassNotFoundException
チェック例外と非チェック例外
特性 チェック例外 非チェック例外
基底クラス Exception のサブクラス
(RuntimeException を除く)		 RuntimeException のサブクラス
および Error のサブクラス
コンパイラのチェック 必須（try-catch または throws 宣言） 任意（チェックされない）
主な用途 回復可能な例外状況
（ファイル操作、ネットワーク通信など）		 プログラミングエラー
（バグや設計ミスに起因する問題）

try-catch-finally構文
try { // 例外が発生する可能性のあるコード FileReader file = new FileReader("sample.txt"); BufferedReader reader = new BufferedReader(file); String line = reader.readLine(); System.out.println(line); } catch (FileNotFoundException e) { // FileNotFoundExceptionが発生した場合の処理 System.err.println("ファイルが見つかりません: " + e.getMessage()); } catch (IOException e) { // その他のIOExceptionが発生した場合の処理 System.err.println("入出力エラーが発生しました: " + e.getMessage()); } finally { // 例外が発生してもしなくても必ず実行される処理 System.out.println("finally ブロックが実行されました"); // リソースのクローズなどの後処理を行う // reader.close(); // 本来はここでクローズ処理を行う }
try-with-resources構文（Java 7以降）

AutoCloseable インターフェースを実装したリソースを自動的にクローズする構文です。

try ( FileReader file = new FileReader("sample.txt"); BufferedReader reader = new BufferedReader(file) ) { String line = reader.readLine(); System.out.println(line); } catch (IOException e) { System.err.println("エラーが発生しました: " + e.getMessage()); e.printStackTrace(); } // リソースは自動的にクローズされます
カスタム例外の作成

特定のアプリケーション要件に合わせたカスタム例外クラスを作成できます。

// チェック例外の作成例 public class InvalidUserDataException extends Exception { private static final long serialVersionUID = 1L; public InvalidUserDataException() { super(); } public InvalidUserDataException(String message) { super(message); } public InvalidUserDataException(String message, Throwable cause) { super(message, cause); } } // 非チェック例外の作成例 public class BusinessLogicException extends RuntimeException { private static final long serialVersionUID = 1L; private String errorCode; public BusinessLogicException(String message) { super(message); } public BusinessLogicException(String message, String errorCode) { super(message); this.errorCode = errorCode; } public String getErrorCode() { return errorCode; } }

パッケージとアクセス制御

パッケージはJavaにおいてクラス、インターフェース、列挙型を整理するための仕組みです。 関連するクラスをグループ化し、名前空間の衝突を避け、アクセス制御を提供します。

パッケージの宣言と構造
// ファイル先頭にパッケージ宣言を記述 package com.example.myproject.util; // インポート宣言 import java.util.List; import java.util.ArrayList; import static java.lang.Math.PI; // static import public class Calculator { // クラスの実装 }

パッケージ名はドメイン名を逆順にした形式が一般的です（例: com.example.project）。 ソースファイルの物理的な配置もパッケージ構造に一致させる必要があります。

アクセス修飾子とパッケージの関係
アクセス修飾子 同じクラス内 同じパッケージ内 サブクラス その他のクラス
private ○ × × ×
デフォルト
(パッケージプライベート)		 ○ ○ ×
(同じパッケージなら○)		 ×
protected ○ ○ ○ ×
public ○ ○ ○ ○
インポート文の種類
  • 単一クラスのインポート: import java.util.ArrayList;
  • パッケージ内のすべてのクラスをインポート: import java.util.*;
  • static インポート（メソッド・フィールド）: import static java.lang.Math.PI;
  • static インポート（すべてのメソッド・フィールド）: import static java.lang.Math.*;
パッケージ設計のベストプラクティス
  • 関連する機能ごとにパッケージを分ける
  • Java の命名規則に従う（小文字のみ）
  • パッケージごとに適切なアクセス制御を設定する
  • 循環依存を避ける
  • 内部実装クラスは公開しない（パッケージプライベートにする）
  • 公開APIとして使用されるクラスのみ public にする
